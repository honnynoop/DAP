# DA 엔터티 정의 과정 — 상세 실무 예시 참고 문서

> **적용 시나리오**: 온라인 쇼핑몰 (B2C 전자상거래) 시스템 구축 프로젝트  
> **목적**: 개념 데이터 모델링 단계에서 엔터티를 식별하고 정의하는 전 과정을 실무 수준으로 예시화  
> **작성 기준**: DA(데이터 아키텍처) 방법론 / DAP 수험 참고용

---

## 프로젝트 배경

| 항목 | 내용 |
|---|---|
| 프로젝트명 | ShopCore 전사 데이터 모델링 |
| 업종 | B2C 온라인 쇼핑몰 |
| 주요 업무 | 회원 관리, 상품 등록/관리, 주문/결제, 배송, 반품/교환, 정산 |
| 모델링 범위 | 개념 → 논리 (본 문서는 개념 단계의 엔터티 정의에 집중) |

## Step 1. 업무 분석 자료 검토

### 1-1. 수집 자료 목록

| 자료 유형 | 자료명 | 수집 방법 |
|---|---|---|
| 업무 기술서 | 쇼핑몰 업무 프로세스 정의서 v1.2 | 현업 담당자 제공 |
| 요구사항 정의서 | 신규 시스템 기능 요구사항 목록 | PM 제공 |
| 현행 화면 | 관리자 페이지 / 고객 화면 캡처 120장 | 현행 시스템 분석 |
| 현행 DB | 기존 MySQL 테이블 DDL 65개 | DB 어드민 제공 |
| 장표 | 주문확인서, 세금계산서, 배송조회 화면 | 운영팀 제공 |
| 인터뷰 결과 | 영업팀·물류팀·CS팀 인터뷰록 | 워크숍 (3회) |

### 1-2. 업무 기술서 핵심 발췌 (예시)

```
[회원 업무]
"고객은 쇼핑몰에 회원으로 가입하여 개인정보를 등록한다.
 회원은 일반/VIP/VVIP 등급으로 구분되며 구매 실적에 따라 등급이 변동된다.
 회원은 여러 개의 배송지 주소를 등록할 수 있다."

[상품 업무]
"판매자(셀러)는 상품을 등록하고 카테고리를 지정한다.
 상품은 색상·사이즈 등 옵션을 가질 수 있으며,
 각 옵션 조합마다 재고가 별도로 관리된다."

[주문 업무]
"회원이 장바구니에 상품을 담고 주문을 요청하면
 주문번호가 생성되고 결제가 진행된다.
 하나의 주문에 여러 상품이 포함될 수 있다."

[배송 업무]
"결제 완료된 주문은 창고에서 출고되어 택배사를 통해 배송된다.
 배송 중 현재 위치 및 상태(집하/이동중/배달완료)가 추적된다."
```

### 3-2. 오류 사례 상세 분석

#### 사례 ① 택배사: 속성으로 전환

```
[오류]  택배사를 독립 엔터티로 정의
        → ENT: 택배사 (택배사코드, 택배사명)
           속성 단 2개, 단순 코드성 데이터

[원인]  현행 DB에 DELIVERY_COMPANY 테이블이 존재하여 그대로 엔터티로 도출

[조치]  배송 엔터티의 속성(택배사코드, 택배사명)으로 이동
        단, 향후 택배사별 SLA·계약 관리가 추가되면 엔터티로 승격 검토
```

#### 사례 ② 포인트: 포인트이력 엔터티로 재정의

```
[오류]  '포인트'를 엔터티로 정의
        → 회원 포인트 잔액만 관리한다면 회원 엔터티의 속성으로 충분

[분석]  인터뷰 결과, 포인트 적립/사용/소멸 이력을 모두 관리해야 한다는 요구 확인

[조치]  '포인트이력' 엔터티로 재정의
        → 적립/차감/소멸 유형, 거래일시, 거래금액, 잔액 등을 관리
           (회원 엔터티에는 현재 포인트 잔액 속성만 유지)
```

#### 사례 ③ 반품/교환: 클레임 엔터티로 통합

```
[오류]  반품, 교환을 각각 별도 엔터티로 도출

[분석]  - 반품: 환불 처리 흐름
        - 교환: 재발송 처리 흐름
        → 접수~처리~완료 프로세스 구조가 동일,
          처리 유형(반품/교환)으로 구분 가능

[조치]  '클레임' 엔터티로 통합, 클레임유형(반품/교환) 속성으로 구분
        → 향후 기타 민원 유형 확장 용이
```

### 📄 엔터티 정의서 #1: 회원 (Member)

| 항목 | 내용 |
|---|---|
| **엔터티 코드** | ENT-001 |
| **엔터티명** | 회원 / Member |
| **주제영역** | 회원 |
| **엔터티 유형** | 기본 엔터티 (Fundamental Entity) |
| **정의** | ShopCore 쇼핑몰에 가입하여 서비스를 이용하는 개인으로, 회원가입 시점부터 탈퇴 시점까지 관리되는 데이터 집합. 비회원 구매는 포함하지 않음. |
| **인스턴스 예시** | MBR-0001: 홍길동, hong@email.com, 010-1234-5678, VIP 등급 |
| | MBR-0002: 김영희, kim@email.com, 010-9876-5432, 일반 등급 |
| **식별자 (후보키)** | 회원번호 (MemberNo) — 시스템 자동 채번 |
| | 이메일 주소 (Email) — 로그인 ID로도 사용, 유일성 보장 |
| **핵심 속성** | 회원번호, 이름, 이메일, 휴대폰번호, 생년월일, 성별, 가입일시, 탈퇴일시, 회원상태(활성/휴면/탈퇴), 포인트잔액 |
| **관련 엔터티** | 회원등급 (회원은 하나의 등급에 속한다) |
| | 배송지 (회원은 여러 배송지를 등록할 수 있다) |
| | 주문 (회원은 여러 주문을 한다) |
| | 포인트이력 (회원은 포인트 적립/사용 이력을 갖는다) |
| **데이터 오너** | 회원서비스팀 |
| **작성자 / 작성일** | 이다현 DA / 2024-03-05 |
| **특이사항** | 소셜 로그인 회원(카카오·네이버)은 동일 엔터티에서 로그인유형 속성으로 구분 |

### 6-1. 관계 정의표

| 2 | 회원 | 등록한다 | 배송지 | 1:N | 선택 | 필수 | 회원 1명이 여러 배송지 등록 가능 |
| 3 | 판매자 | 등록한다 | 상품 | 1:N | 선택 | 필수 | 상품은 반드시 판매자가 있어야 함 |
| 4 | 카테고리 | 분류한다 | 상품 | 1:N | 선택 | 필수 | 상품은 반드시 카테고리에 속함 |

| 14 | 회원 | 작성한다 | 리뷰 | 1:N | 선택 | 필수 | 리뷰는 반드시 회원이 작성 |
| 15 | 판매자 | 정산받는다 | 정산 | 1:N | 선택 | 필수 | 정산은 판매자별 생성 |

  ├── 한다(1:N) ──── [주문] ──── 포함한다(1:N) ──── [주문상세]
