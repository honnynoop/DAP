## 1. 전사아키텍처 개요

### 1.1 정의

**전사아키텍처(EA)**는 조직의 비즈니스 목표를 달성하기 위해 비즈니스, 데이터, 애플리케이션, 기술 인프라를 통합적으로 설계하고 관리하는 체계입니다.

### 1.2 목적

| 목적 | 설명 |
|------|------|
| **전략적 정렬** | IT와 비즈니스 전략의 일치 |
| **중복 제거** | 시스템 및 데이터 중복 최소화 |
| **비용 절감** | IT 투자 최적화 및 운영 효율성 향상 |
| **민첩성 향상** | 변화에 대한 빠른 대응 능력 확보 |
| **리스크 관리** | IT 리스크 식별 및 완화 |
| **의사결정 지원** | 데이터 기반 의사결정 기반 마련 |

### 1.3 EA의 필요성

```
비즈니스 환경 변화
    ↓
디지털 트랜스포메이션 요구
    ↓
복잡한 IT 환경
    ↓
통합적 관리 필요
    ↓
전사아키텍처 구축
```

#### 2.2.2 TOGAF ADM 단계별 설명

| 단계 | 명칭 | 주요 활동 | 산출물 |
|------|------|----------|--------|
| **Preliminary** | 준비 | EA 프레임워크 및 원칙 정의 | EA 거버넌스 프레임워크 |
| **A** | Architecture Vision | 비전 및 범위 정의 | 아키텍처 비전 문서 |
| **B** | Business Architecture | 비즈니스 프로세스 및 조직 설계 | 비즈니스 아키텍처 문서 |
| **C** | Data Architecture | 데이터 구조 및 관리 체계 설계 | 데이터 아키텍처 문서 |
| **D** | Application Architecture | 애플리케이션 포트폴리오 설계 | 애플리케이션 아키텍처 문서 |
| **E** | Technology Architecture | 기술 인프라 설계 | 기술 아키텍처 문서 |
| **F** | Opportunities & Solutions | 구현 프로젝트 식별 | 구현 및 마이그레이션 계획 |
| **G** | Migration Planning | 세부 마이그레이션 계획 수립 | 마이그레이션 로드맵 |
| **H** | Implementation Governance | 구현 관리 및 통제 | 아키텍처 계약 |
| **Requirements** | Requirements Management | 요구사항 지속 관리 | 요구사항 저장소 |

### 2.3 Zachman Framework

#### 2.3.1 Zachman 프레임워크 구조

| 관점(Row) \ 질문(Column) | What (데이터) | How (기능) | Where (네트워크) | Who (사람) | When (시간) | Why (동기) |
|-------------------------|------------|----------|---------------|----------|----------|----------|
| **Scope (맥락가)** | 주요 데이터 목록 | 비즈니스 프로세스 목록 | 비즈니스 위치 목록 | 주요 조직 목록 | 주요 이벤트 목록 | 비즈니스 목표 목록 |
| **Business Model (소유자)** | 개념적 데이터 모델 | 비즈니스 프로세스 모델 | 비즈니스 물류 시스템 | 업무 흐름 모델 | 마스터 일정 | 비즈니스 계획 |
| **System Model (설계자)** | 논리적 데이터 모델 | 애플리케이션 아키텍처 | 분산 시스템 아키텍처 | 인터페이스 아키텍처 | 프로세스 구조 | 비즈니스 규칙 모델 |
| **Technology Model (빌더)** | 물리적 데이터 모델 | 시스템 설계 | 기술 아키텍처 | 프레젠테이션 아키텍처 | 제어 구조 | 규칙 설계 |
| **Detailed Representations (하청업체)** | 데이터 정의 | 프로그램 | 네트워크 아키텍처 | 보안 아키텍처 | 타이밍 정의 | 규칙 명세 |
| **Functioning Enterprise (사용자)** | 실제 데이터 | 실행 프로그램 | 통신 네트워크 | 조직 | 일정 | 전략 |

                 │

### 3.2 비즈니스 아키텍처 (Business Architecture)

| 구성요소 | 설명 | 주요 산출물 |
|---------|------|-----------|
| **비즈니스 전략** | 조직의 목표 및 전략 방향 | 전략 맵, BSC |
| **비즈니스 프로세스** | 업무 처리 절차 및 흐름 | BPMN, 프로세스 맵 |

#### 3.3.1 데이터 아키텍처 구성요소

```
┌─────────────────────────────────────────┐
│         데이터 표준 (Standards)           │
│  - 표준 단어                              │
│  - 표준 도메인                            │
│  - 표준 용어                              │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│      데이터 모델 (Data Model)             │
│  - 개념 데이터 모델 (CDM)                  │
│  - 논리 데이터 모델 (LDM)                  │
│  - 물리 데이터 모델 (PDM)                  │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│    데이터 흐름 (Data Flow)                │
│  - 데이터 인터페이스                       │
│  - ETL/ELT 프로세스                       │
│  - 데이터 통합                            │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│  데이터 저장소 (Data Repository)          │
│  - 운영 DB                                │

│  - 데이터 레이크                          │
└─────────────────────────────────────────┘
```

#### 3.3.2 데이터 모델링 수준

| 모델 단계 | 목적 | 주요 내용 | 표기법 |
|----------|------|----------|--------|
| **개념 데이터 모델 (CDM)** | 업무 관점의 데이터 구조 파악 | 핵심 엔티티 및 관계 | ERD (간략) |
| **논리 데이터 모델 (LDM)** | 상세 업무 규칙 반영 | 모든 엔티티, 속성, 관계, 정규화 | ERD (상세) |
| **물리 데이터 모델 (PDM)** | 실제 DB 구현 | 테이블, 컬럼, 인덱스, 파티션 | DDL |

#### 3.3.3 데이터 표준화 체계

```
┌──────────────────┐
│   표준 단어       │  ← 기본 단어 정의 (예: 고객, 주문, 금액)
└────────┬─────────┘
         │
┌────────▼─────────┐
│   표준 도메인     │  ← 데이터 타입 및 형식 (예: 금액_10,2)
└────────┬─────────┘
         │
┌────────▼─────────┐
│   표준 용어       │  ← 업무 용어 (표준단어 + 도메인)
└────────┬─────────┘
         │
┌────────▼─────────┐
│   표준 코드       │  ← 코드 체계 및 값
└──────────────────┘
```

### 3.4 애플리케이션 아키텍처 (Application Architecture)

#### 3.4.1 애플리케이션 구성도

```
┌─────────────────────────────────────────────────┐
│              Presentation Layer                 │
│  (Web, Mobile, Desktop 사용자 인터페이스)         │
└──────────────────┬──────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────┐
│             Business Logic Layer                │
│  (비즈니스 규칙, 워크플로우, 트랜잭션 처리)        │
└──────────────────┬──────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────┐
│             Integration Layer                   │
│  (ESB, API Gateway, 메시지 큐)                   │
└──────────────────┬──────────────────────────────┘

│               Data Layer                        │

```

#### 3.4.2 애플리케이션 통합 방식

| 통합 방식 | 설명 | 장점 | 단점 | 사용 사례 |
|----------|------|------|------|----------|
| **Point-to-Point** | 직접 연결 | 구현 간단, 빠른 속도 | 복잡도 증가, 유지보수 어려움 | 소규모 시스템 |
| **ESB** | 중앙 통합 버스 | 표준화, 느슨한 결합 | 단일 장애점, 성능 병목 | 대규모 엔터프라이즈 |
| **API Gateway** | API 관리 플랫폼 | 보안, 모니터링, 버전관리 | 추가 계층 | 마이크로서비스 |
| **Message Queue** | 비동기 메시징 | 확장성, 안정성 | 복잡한 설계 | 비동기 처리 |
| **EAI** | 엔터프라이즈 통합 | 다양한 어댑터 제공 | 고비용 | 레거시 통합 |

## 4. EA 개발 방법론

### 4.1 개발 접근법

| 접근법 | 설명 | 장점 | 단점 |
|--------|------|------|------|
| **Top-Down** | 전략에서 시작하여 세부로 | 전략 정렬 우수 | 시간 소요, 초기 비용 높음 |
| **Bottom-Up** | 현재 시스템에서 시작 | 빠른 시작, 현실적 | 전략 정렬 부족 |
| **Middle-Out** | 핵심 영역 중심 | 균형적 접근 | 조정 노력 필요 |

### 4.2 EA 개발 프로세스

```
1. 현황 분석 (As-Is)
   ├─ 비즈니스 현황
   ├─ 데이터 현황
   ├─ 애플리케이션 현황
   └─ 기술 현황
        ↓
2. 목표 모델 수립 (To-Be)
   ├─ 목표 비즈니스 아키텍처
   ├─ 목표 데이터 아키텍처
   ├─ 목표 애플리케이션 아키텍처
   └─ 목표 기술 아키텍처
        ↓
3. 갭 분석 (Gap Analysis)
   ├─ 현황과 목표 비교
   ├─ 개선 사항 도출
   └─ 우선순위 결정
        ↓
4. 마이그레이션 계획
   ├─ 단계별 이행 계획
   ├─ 프로젝트 정의
   └─ 리소스 계획
        ↓
5. 구현 및 관리
   ├─ 프로젝트 실행
   ├─ 아키텍처 준수 관리
   └─ 지속적 개선
```

### 4.3 성과 측정 지표

| 지표 유형 | 측정 항목 | 설명 |

## 마무리

전사아키텍처는 조직의 디지털 혁신을 위한 필수 요소입니다. 성공적인 EA 구축을 위해서는:

1. **경영진의 지원**: 최고 경영진의 관심과 지원
2. **명확한 목표**: 비즈니스 목표와 정렬된 EA 목표
3. **단계적 접근**: 작은 성공을 통한 점진적 발전
4. **지속적 관리**: EA는 일회성이 아닌 지속적 활동
5. **문화 조성**: EA 가치에 대한 조직 전체의 이해

EA는 기술이 아닌 **비즈니스 혁신을 위한 전략적 도구**임을 기억해야 합니다.
