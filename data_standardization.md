## 1. 데이터 표준화 개요

### 1.1 데이터 표준화 정의

**데이터 표준화**는 조직 내에서 사용되는 데이터의 명칭, 정의, 형식, 규칙 등을 통일하여 데이터의 일관성, 정확성, 활용성을 높이는 활동입니다.

### 1.2 데이터 표준화의 필요성

| 문제점 (Before) | 해결책 (After) | 효과 |
|----------------|---------------|------|
| 동일 데이터의 다른 명칭 | 표준 용어 사용 | 의사소통 명확화 |
| 데이터 중복 및 불일치 | 표준 데이터 모델 | 데이터 품질 향상 |
| 시스템 간 연계 어려움 | 표준 인터페이스 | 통합 용이성 증대 |
| 데이터 의미 불명확 | 표준 메타데이터 | 데이터 이해도 향상 |
| 개발 및 유지보수 비용 증가 | 표준 재사용 | 비용 절감 |

### 1.3 데이터 표준화의 목적

```
┌────────────────────────────────────────┐
│         데이터 표준화 목적              │
├────────────────────────────────────────┤
│                                        │
│  ┌──────────────┐  ┌──────────────┐   │
│  │  데이터 품질  │  │  데이터 통합  │   │
│  │    향상      │  │    용이      │   │
│  └──────────────┘  └──────────────┘   │
│                                        │
│  ┌──────────────┐  ┌──────────────┐   │
│  │ 의사소통 개선 │  │  개발 효율성  │   │
│  │              │  │    향상      │   │
│  └──────────────┘  └──────────────┘   │
│                                        │
│  ┌──────────────┐  ┌──────────────┐   │
│  │  규제 준수   │  │  재사용성    │   │
│  │              │  │    증대      │   │
│  └──────────────┘  └──────────────┘   │
└────────────────────────────────────────┘
```

### 1.4 데이터 표준화의 효과

| 영역 | 효과 | 측정 지표 |
|------|------|----------|
| **품질** | 데이터 정확성 향상 | 데이터 오류율 감소 30% |
| **비용** | 개발/유지보수 비용 절감 | 개발 기간 단축 40% |
| **통합** | 시스템 통합 용이 | 통합 시간 단축 50% |
| **활용** | 데이터 재사용성 증대 | 재사용률 60% 이상 |
| **규제** | 컴플라이언스 준수 | 감사 지적 사항 감소 |

## 2. 데이터 표준 구성요소

### 2.1 데이터 표준 체계도

```
┌──────────────────────────────────────────────┐
│              데이터 표준 체계                  │
└──────────────────┬───────────────────────────┘
                   │
    ┌──────────────┼──────────────┐
    │              │              │
┌───▼────┐    ┌───▼────┐    ┌───▼────┐
│표준단어 │    │표준도메인│   │표준용어 │
└───┬────┘    └───┬────┘    └───┬────┘
    │              │              │
    │         ┌────▼────┐         │
    │         │표준코드  │         │
    │         └────┬────┘         │
    │              │              │
    └──────────────┼──────────────┘
                   │
            ┌──────▼──────┐
            │ 메타데이터   │
            └─────────────┘
```

### 2.2 표준 단어 (Standard Word)

#### 2.2.1 정의
표준 단어는 데이터를 표현하는 **최소 단위의 기본 단어**로, 모든 표준 용어의 기초가 됩니다.

#### 2.2.2 표준 단어 속성

| 속성 | 설명 | 예시 |
|------|------|------|
| **단어명** | 표준화된 단어 | 고객 |
| **영문명** | 영문 표기 | CUSTOMER |
| **영문약어명** | 영문 약어 | CUST |
| **정의** | 단어의 의미 | 상품 또는 서비스를 구매하는 개인 또는 법인 |
| **동의어** | 같은 의미의 다른 단어 | 거래처, 클라이언트 |
| **금칙어** | 사용 금지 단어 | 손님, 회원 |

#### 2.2.3 표준 단어 예시

| 단어명 | 영문명 | 영문약어 | 정의 | 동의어 |
|--------|--------|---------|------|--------|
| 고객 | CUSTOMER | CUST | 상품/서비스를 구매하는 개인/법인 | 거래처 |
| 주문 | ORDER | ORD | 고객이 상품 구매를 요청하는 행위 | 발주 |
| 금액 | AMOUNT | AMT | 화폐로 표시된 가치 | 가격 |
| 일자 | DATE | DT | 특정 날짜 | 날짜 |
| 코드 | CODE | CD | 데이터를 분류하기 위한 기호 | 부호 |
| 명 | NAME | NM | 사물의 이름 | 성명 |
| 번호 | NUMBER | NO | 순서나 식별을 위한 숫자 | 넘버 |
| 구분 | DIVISION | DIV | 종류를 나누는 기준 | 분류 |
| 여부 | WHETHER | YN | 그러한지 아닌지를 나타냄 | - |
| 수량 | QUANTITY | QTY | 물건의 개수나 분량 | - |

#### 2.2.4 표준 단어 분류

```
표준 단어
├─ 엔티티 단어
│  ├─ 고객, 주문, 상품, 직원
│  └─ 계좌, 거래, 프로젝트
├─ 속성 단어
│  ├─ 명, 번호, 코드, 일자
│  └─ 금액, 수량, 구분, 여부
└─ 수식 단어
   ├─ 시작, 종료, 최초, 최종
   └─ 총, 평균, 최대, 최소
```

### 2.3 표준 도메인 (Standard Domain)

#### 2.3.1 정의
표준 도메인은 **데이터의 타입, 길이, 형식을 정의**한 것으로, 동일한 성격의 데이터에 일관된 형식을 부여합니다.

#### 2.3.2 표준 도메인 속성

| 속성 | 설명 | 예시 |
|------|------|------|
| **도메인명** | 도메인 이름 | 금액 |
| **데이터타입** | 데이터 유형 | NUMBER, VARCHAR2, DATE |
| **길이** | 최대 길이 | 10,2 (정수 8자리, 소수 2자리) |
| **NULL 허용** | NULL 값 허용 여부 | Y, N |
| **기본값** | 초기 기본값 | 0, SYSDATE |

#### 2.3.3 표준 도메인 예시

| 도메인명 | 데이터타입 | 길이 | 설명 | 예시 |
|---------|-----------|------|------|------|
| **금액** | NUMBER | 15,2 | 화폐 금액 | 1234567.89 |
| **수량** | NUMBER | 10,2 | 물품 수량 | 100.50 |
| **비율** | NUMBER | 5,2 | 백분율 | 12.34 |
| **일자** | DATE | - | 날짜 | 2024-02-10 |
| **일시** | TIMESTAMP | - | 날짜+시간 | 2024-02-10 14:30:00 |
| **코드_2** | VARCHAR2 | 2 | 2자리 코드 | 01, AB |
| **코드_4** | VARCHAR2 | 4 | 4자리 코드 | A001 |
| **명_100** | VARCHAR2 | 100 | 이름 (100자) | 홍길동 |
| **명_200** | VARCHAR2 | 200 | 이름 (200자) | 주식회사 한국데이터 |
| **번호_13** | VARCHAR2 | 13 | 번호 (13자리) | 1234567890123 |
| **여부** | CHAR | 1 | Y/N 플래그 | Y, N |
| **설명_1000** | VARCHAR2 | 1000 | 설명 (1000자) | 상세 설명 내용 |
| **텍스트** | CLOB | - | 대용량 텍스트 | 긴 텍스트 |
| **이미지** | BLOB | - | 이진 데이터 | 이미지 파일 |

#### 2.3.4 도메인 명명 규칙

```
도메인명 = 데이터유형_길이

예시:
- 금액_15,2    → NUMBER(15,2)
- 명_100       → VARCHAR2(100)
- 코드_2       → VARCHAR2(2)
- 일자         → DATE
- 여부         → CHAR(1)
```

### 2.4 표준 용어 (Standard Term)

#### 2.4.1 정의
표준 용어는 **표준 단어를 조합하여 만든 업무상 사용되는 용어**로, 실제 테이블의 컬럼명으로 사용됩니다.

#### 2.4.2 표준 용어 생성 규칙

```
표준 용어 = [수식어] + 대표단어 + [분류어] + 속성어

예시:
고객ID        = 고객 + ID
주문일자      = 주문 + 일자
총주문금액    = 총 + 주문 + 금액
상품분류코드  = 상품 + 분류 + 코드
시작일시      = 시작 + 일시
최종변경일자  = 최종 + 변경 + 일자
```

#### 2.4.3 표준 용어 속성

| 속성 | 설명 | 예시 |
|------|------|------|
| **용어명** | 표준화된 용어 | 고객명 |
| **영문명** | 영문 표기 | CUSTOMER_NAME |
| **영문약어명** | 영문 약어 | CUST_NM |
| **도메인** | 할당된 도메인 | 명_100 |
| **정의** | 용어의 의미 | 고객의 이름 |
| **사용 테이블** | 용어가 사용되는 테이블 | 고객, 주문 |

#### 2.4.4 표준 용어 예시

| 용어명 | 영문명 | 영문약어 | 도메인 | 정의 |
|--------|--------|---------|--------|------|
| 고객ID | CUSTOMER_ID | CUST_ID | 번호_10 | 고객을 식별하는 고유번호 |
| 고객명 | CUSTOMER_NAME | CUST_NM | 명_100 | 고객의 이름 |
| 주문번호 | ORDER_NUMBER | ORD_NO | 번호_20 | 주문을 식별하는 고유번호 |
| 주문일자 | ORDER_DATE | ORD_DT | 일자 | 주문이 발생한 날짜 |
| 주문금액 | ORDER_AMOUNT | ORD_AMT | 금액 | 주문의 총 금액 |
| 상품코드 | PRODUCT_CODE | PROD_CD | 코드_10 | 상품을 분류하는 코드 |
| 수량 | QUANTITY | QTY | 수량 | 주문한 상품의 개수 |
| 할인율 | DISCOUNT_RATE | DISC_RT | 비율 | 할인되는 비율 (%) |
| 배송상태코드 | DELIVERY_STATUS_CODE | DLVY_STAT_CD | 코드_2 | 배송 상태 (01:준비중, 02:배송중, 03:완료) |
| 등록일시 | REGISTRATION_DATETIME | REG_DTM | 일시 | 데이터가 등록된 날짜와 시간 |
| 사용여부 | USE_WHETHER | USE_YN | 여부 | 사용 가능 여부 (Y:사용, N:미사용) |

### 2.5 표준 코드 (Standard Code)

#### 2.5.1 정의
표준 코드는 **데이터를 분류하고 식별하기 위한 기호 체계**입니다.

#### 2.5.2 코드 분류

```
표준 코드
├─ 단일 코드 (Single Code)
│  └─ 하나의 값만 가지는 코드
│     예) 성별코드: M(남성), F(여성)
│
├─ 그룹 코드 (Group Code)
│  └─ 여러 하위 코드를 가지는 상위 코드
│     예) 주문상태: 01(주문접수), 02(결제완료), 03(배송중)
│
└─ 계층 코드 (Hierarchical Code)
   └─ 계층 구조를 가지는 코드
      예) 상품분류: 01(전자제품) - 0101(TV) - 010101(OLED TV)
```

#### 2.5.3 표준 코드 속성

| 속성 | 설명 | 예시 |
|------|------|------|
| **코드ID** | 코드 식별자 | ORDER_STATUS_CODE |
| **코드명** | 코드 이름 | 주문상태코드 |
| **코드값** | 실제 코드 값 | 01 |
| **코드값명** | 코드값의 의미 | 주문접수 |
| **정의** | 코드의 설명 | 주문이 접수된 상태 |
| **정렬순서** | 표시 순서 | 1 |
| **사용여부** | 사용 가능 여부 | Y |

#### 2.5.4 표준 코드 예시

**성별코드 (GENDER_CODE)**
| 코드값 | 코드값명 | 설명 |
|--------|---------|------|
| M | 남성 | Male |
| F | 여성 | Female |

**주문상태코드 (ORDER_STATUS_CODE)**
| 코드값 | 코드값명 | 설명 |
|--------|---------|------|
| 01 | 주문접수 | 주문이 접수됨 |
| 02 | 결제완료 | 결제가 완료됨 |
| 03 | 상품준비 | 배송을 위해 상품 준비 중 |
| 04 | 배송중 | 배송이 진행 중 |
| 05 | 배송완료 | 배송이 완료됨 |
| 06 | 주문취소 | 주문이 취소됨 |
| 07 | 환불완료 | 환불이 완료됨 |

**상품분류코드 (PRODUCT_CATEGORY_CODE) - 계층형**
| 코드값 | 코드값명 | 상위코드 | 레벨 |
|--------|---------|---------|------|
| 01 | 전자제품 | NULL | 1 |
| 0101 | TV | 01 | 2 |
| 010101 | OLED TV | 0101 | 3 |
| 010102 | LED TV | 0101 | 3 |
| 0102 | 냉장고 | 01 | 2 |
| 02 | 의류 | NULL | 1 |
| 0201 | 남성의류 | 02 | 2 |
| 0202 | 여성의류 | 02 | 2 |

#### 2.5.5 코드 설계 원칙

| 원칙 | 설명 | 좋은 예 | 나쁜 예 |
|------|------|---------|---------|
| **유일성** | 중복되지 않는 고유값 | 각 코드값이 유일 | 동일 코드값 존재 |
| **간결성** | 가능한 짧게 | 01, 02 | ORDER_RECEIVED |
| **확장성** | 추가 코드 여유 | 01~99 (100개 가능) | 1~9 (10개만 가능) |
| **의미성** | 의미 파악 가능 | 01(주문), 02(배송) | A1, B2 |
| **일관성** | 동일한 규칙 적용 | 모두 2자리 숫자 | 01, A, 100 혼재 |

## 3. 데이터 표준화 프로세스

### 3.1 데이터 표준화 절차

```
1단계: 현황 분석
   ├─ 기존 데이터 조사
   ├─ 용어 수집
   └─ 문제점 파악
        ↓
2단계: 표준 정의
   ├─ 표준 단어 정의
   ├─ 표준 도메인 정의
   ├─ 표준 용어 정의
   └─ 표준 코드 정의
        ↓
3단계: 표준 검증
   ├─ 표준 검토
   ├─ 중복 확인
   └─ 승인
        ↓
4단계: 표준 적용
   ├─ 데이터 모델 적용
   ├─ 데이터베이스 적용
   └─ 애플리케이션 적용
        ↓
5단계: 표준 관리
   ├─ 표준 준수 모니터링
   ├─ 표준 변경 관리
   └─ 교육 및 확산
```

### 3.2 단계별 상세 활동

#### 3.2.1 현황 분석 (As-Is Analysis)

| 활동 | 목적 | 산출물 |
|------|------|--------|
| **데이터 자산 조사** | 기존 데이터베이스, 문서 파악 | 데이터 현황 목록 |
| **용어 수집** | 사용 중인 용어 수집 | 용어 수집 목록 |
| **동의어 분석** | 동일 의미의 다른 용어 식별 | 동의어 매핑표 |
| **문제점 도출** | 표준화 필요 영역 파악 | 문제점 분석 보고서 |

**용어 수집 예시**
| 시스템 | 사용 용어 | 의미 | 비고 |
|--------|----------|------|------|
| CRM | 고객명 | 고객 이름 | - |
| ERP | 거래처명 | 고객 이름 | 동일 의미 |
| SCM | 클라이언트명 | 고객 이름 | 동일 의미 |
| CRM | CUST_NAME | 고객 이름 | 영문 표기 불일치 |
| ERP | CUSTOMER_NM | 고객 이름 | 영문 표기 불일치 |

#### 3.2.2 표준 정의 (Standard Definition)

**표준 단어 정의 프로세스**
```
1. 수집된 용어 분석
   ↓
2. 단어 분해 (토큰화)
   예) "총주문금액" → "총" + "주문" + "금액"
   ↓
3. 표준 단어 후보 선정
   ↓
4. 동의어/유사어 그룹핑
   예) 고객 = 거래처 = 클라이언트
   ↓
5. 대표 단어 선정
   예) "고객"을 표준으로 선정
   ↓
6. 표준 단어 등록
```

**표준 용어 정의 프로세스**
```
1. 표준 단어 조합
   예) "고객" + "명" → "고객명"
   ↓
2. 도메인 할당
   예) "고객명" → "명_100" (VARCHAR2(100))
   ↓
3. 영문명 생성
   예) "고객명" → "CUSTOMER_NAME"
   ↓
4. 영문 약어명 생성
   예) "CUSTOMER_NAME" → "CUST_NM"
   ↓
5. 정의 작성
   예) "고객의 이름"
   ↓
6. 표준 용어 등록
```

#### 3.2.3 표준 검증 (Validation)

| 검증 항목 | 검증 내용 | 도구/방법 |
|----------|----------|----------|
| **중복 검사** | 동일 용어 중복 등록 확인 | 자동 검증 도구 |
| **명명 규칙 준수** | 표준 명명 규칙 준수 여부 | 규칙 체크리스트 |
| **도메인 일관성** | 동일 의미 용어의 도메인 일치 | 도메인 매핑 검증 |
| **영문명 검증** | 영문명/약어 규칙 준수 | 영문명 검증 도구 |
| **업무 적합성** | 업무 의미와 정의 일치 | 업무 전문가 검토 |

#### 3.2.4 표준 적용 (Application)

**데이터 모델 적용**
```sql
-- Before (비표준)
CREATE TABLE CUSTOMER_INFO (
    CUST_NO VARCHAR2(20),      -- 불일치: 번호 vs NO
    NAME VARCHAR2(50),          -- 불일치: 명 누락
    REG_DATE DATE,              -- 불일치: 일자 vs DATE
    ACTIVE CHAR(1)              -- 불일치: 여부 누락
);

-- After (표준 적용)
CREATE TABLE 고객 (
    고객번호 VARCHAR2(20),      -- 표준 용어
    고객명 VARCHAR2(100),        -- 표준 용어 + 표준 도메인
    등록일자 DATE,               -- 표준 용어
    사용여부 CHAR(1)             -- 표준 용어
);
```

#### 3.2.5 표준 관리 (Management)

```
지속적인 표준 관리
├─ 표준 준수 모니터링
│  ├─ 주기적 점검 (월/분기)
│  ├─ 위반 사항 적발
│  └─ 시정 조치
├─ 표준 변경 관리
│  ├─ 변경 요청 접수
│  ├─ 영향도 분석
│  ├─ 승인/반려
│  └─ 변경 이력 관리
└─ 교육 및 확산
   ├─ 신규 직원 교육
   ├─ 정기 교육
   └─ 표준 가이드 배포
```

## 5. 메타데이터 관리

### 5.1 메타데이터 정의

**메타데이터(Metadata)**는 "데이터에 대한 데이터"로, 데이터의 의미, 구조, 특성을 설명하는 정보입니다.

### 5.2 메타데이터 분류

```
메타데이터
├─ 기술 메타데이터 (Technical Metadata)
│  ├─ 테이블 정보
│  ├─ 컬럼 정보
│  ├─ 데이터 타입
│  ├─ 인덱스 정보
│  └─ 제약조건
│
├─ 업무 메타데이터 (Business Metadata)
│  ├─ 용어 정의
│  ├─ 업무 규칙
│  ├─ 데이터 소유자
│  └─ 데이터 사용처
│
└─ 운영 메타데이터 (Operational Metadata)
   ├─ 데이터 품질 지표
   ├─ 접근 권한
   ├─ 변경 이력
   └─ 사용 통계
```

### 5.3 메타데이터 저장소 (Metadata Repository)

#### 5.3.1 메타데이터 모델

**테이블 메타데이터**
```sql
CREATE TABLE META_TABLE (
    테이블ID VARCHAR2(50) PRIMARY KEY,
    테이블물리명 VARCHAR2(100),
    테이블논리명 VARCHAR2(200),
    업무영역 VARCHAR2(50),
    테이블설명 VARCHAR2(1000),
    데이터소유자 VARCHAR2(50),
    등록일시 TIMESTAMP,
    변경일시 TIMESTAMP
);
```

**컬럼 메타데이터**
```sql
CREATE TABLE META_COLUMN (
    컬럼ID VARCHAR2(50) PRIMARY KEY,
    테이블ID VARCHAR2(50),
    컬럼물리명 VARCHAR2(100),
    컬럼논리명 VARCHAR2(200),
    데이터타입 VARCHAR2(50),
    데이터길이 NUMBER,
    NULL허용여부 CHAR(1),
    기본값 VARCHAR2(100),
    컬럼설명 VARCHAR2(1000),
    도메인ID VARCHAR2(50),
    FOREIGN KEY (테이블ID) REFERENCES META_TABLE(테이블ID),
    FOREIGN KEY (도메인ID) REFERENCES META_DOMAIN(도메인ID)
);
```

**표준 용어 메타데이터**
```sql
CREATE TABLE META_TERM (
    용어ID VARCHAR2(50) PRIMARY KEY,
    용어명 VARCHAR2(100),
    영문명 VARCHAR2(100),
    영문약어명 VARCHAR2(50),
    용어정의 VARCHAR2(1000),
    도메인ID VARCHAR2(50),
    사용여부 CHAR(1),
    등록일시 TIMESTAMP,
    FOREIGN KEY (도메인ID) REFERENCES META_DOMAIN(도메인ID)
);
```

### 5.4 메타데이터 활용

| 활용 목적 | 내용 | 효과 |
|----------|------|------|
| **데이터 이해** | 데이터 의미 파악 | 학습 시간 단축 |
| **영향도 분석** | 변경 시 영향 범위 파악 | 리스크 감소 |
| **데이터 추적** | 데이터 출처 및 흐름 파악 | 데이터 거버넌스 강화 |
| **표준 준수** | 표준 용어 사용 확인 | 품질 향상 |
| **자동화** | 코드 자동 생성 | 개발 생산성 향상 |

## 6. 데이터 표준 관리 체계

### 6.1 데이터 표준 관리 조직

```
┌──────────────────────────────────────┐
│      데이터 표준 관리 위원회          │
│   (의사결정, 승인, 정책 수립)          │
└────────────┬─────────────────────────┘
             │
┌────────────▼─────────────────────────┐
│      데이터 관리 조직 (DA팀)          │
│   (표준 수립, 검토, 관리)              │
└────────────┬─────────────────────────┘
             │
┌────────────▼─────────────────────────┐
│      데이터 모델러 / DBA              │
│   (표준 적용, 모델링)                  │
└────────────┬─────────────────────────┘
             │
┌────────────▼─────────────────────────┐
│      개발자 / 사용자                  │
│   (표준 준수, 피드백)                  │
└──────────────────────────────────────┘
```

### 6.2 역할 및 책임 (RACI)

| 활동 | DA팀 | 모델러 | DBA | 개발자 |
|------|------|--------|-----|--------|
| **표준 정의** | R/A | C | C | I |
| **표준 검토** | A | R | R | C |
| **표준 승인** | A | I | I | I |
| **표준 적용** | C | R | R | R |
| **준수 모니터링** | A/R | C | C | C |
| **표준 교육** | R/A | I | I | I |

*R: Responsible (실행), A: Accountable (책임), C: Consulted (협의), I: Informed (정보제공)*

### 6.3 표준 변경 관리 프로세스

```
1. 변경 요청
   ├─ 요청자: 개발자, 업무담당자
   ├─ 요청 사유 기술
   └─ 제안 내용 작성
        ↓
2. 영향도 분석
   ├─ 사용 현황 조사
   ├─ 영향받는 시스템 파악
   └─ 비용/시간 산정
        ↓
3. 검토 및 승인
   ├─ DA팀 검토
   ├─ 표준위원회 심의
   └─ 승인/반려 결정
        ↓
4. 표준 변경
   ├─ 표준 메타데이터 수정
   ├─ 변경 이력 기록
   └─ 공지
        ↓
5. 적용 및 확산
   ├─ 신규 프로젝트 적용
   ├─ 기존 시스템 단계적 적용
   └─ 교육 실시
```

### 6.4 표준 준수 점검

#### 6.4.1 점검 항목

| 점검 영역 | 점검 내용 | 점검 주기 |
|----------|----------|----------|
| **명명 규칙** | 테이블/컬럼명 표준 준수 | 월 1회 |
| **데이터 타입** | 표준 도메인 사용 여부 | 월 1회 |
| **코드 사용** | 표준 코드 사용 여부 | 분기 1회 |
| **메타데이터** | 메타데이터 정합성 | 분기 1회 |
| **문서화** | 표준 문서 최신화 | 반기 1회 |

#### 6.4.2 점검 도구

```sql
-- 비표준 컬럼명 점검
SELECT TABLE_NAME, COLUMN_NAME
FROM USER_TAB_COLUMNS
WHERE COLUMN_NAME NOT IN (
    SELECT 용어물리명 FROM META_TERM
);

-- 비표준 도메인 사용 점검
SELECT TABLE_NAME, COLUMN_NAME, DATA_TYPE
FROM USER_TAB_COLUMNS
WHERE (DATA_TYPE, DATA_LENGTH) NOT IN (
    SELECT 데이터타입, 데이터길이 FROM META_DOMAIN
);

-- 테이블 명명 규칙 위반 점검
SELECT TABLE_NAME
FROM USER_TABLES
WHERE TABLE_NAME NOT LIKE 'TB_%'
  AND TABLE_NAME NOT LIKE 'TH_%'
  AND TABLE_NAME NOT LIKE 'TT_%';
```

### 6.5 표준 준수율 관리

#### 6.5.1 준수율 계산

```
표준 준수율 = (표준 준수 항목 수 / 전체 점검 항목 수) × 100

예시:
- 전체 컬럼 수: 1,000개
- 표준 용어 사용 컬럼: 850개
- 표준 준수율 = (850 / 1,000) × 100 = 85%
```

#### 6.5.2 목표 관리

| 등급 | 준수율 | 평가 | 조치 |
|------|--------|------|------|
| **우수** | 95% 이상 | 매우 양호 | 현상 유지 |
| **양호** | 85% ~ 94% | 양호 | 개선 권고 |
| **보통** | 70% ~ 84% | 보통 | 개선 필요 |
| **미흡** | 70% 미만 | 미흡 | 즉시 개선 |

#### 8.1.1 Case 1: 테이블 표준화

```sql
CREATE TABLE cust_info (
```

**After (표준 적용)**
```sql
-- 표준 명명 규칙 적용
CREATE TABLE TB_CRM_고객 (
    고객ID VARCHAR2(20) PRIMARY KEY,
    고객명 VARCHAR2(100) NOT NULL,
    전화번호 VARCHAR2(20),

CREATE INDEX TB_CRM_고객_IX_01 ON TB_CRM_고객(고객명);
```

**개선 효과**
- 명명 규칙 통일로 이해도 향상
- 표준 도메인 적용으로 일관성 확보
- 유지보수 용이성 증대

#### 8.1.2 Case 2: 코드 표준화

**Before (하드코딩)**
```sql
-- 주문 상태를 숫자로 관리
CREATE TABLE 주문 (
    주문ID VARCHAR2(20),
    주문상태 NUMBER(1)  -- 1:접수, 2:완료, 3:취소
);

-- 애플리케이션에서 하드코딩
if (주문상태 == 1) {
    // 접수 처리

    // 완료 처리
}
```

**After (표준 코드 적용)**
```sql
-- 표준 코드 테이블
CREATE TABLE TC_공통코드 (
    코드그룹ID VARCHAR2(20),

    주문ID VARCHAR2(20),
```

**개선 효과**

## 마무리

데이터 표준화는 **데이터 품질과 활용성을 높이는 핵심 활동**입니다.

성공적인 데이터 표준화를 위한 핵심 요소:

1. **명확한 표준 정의**: 표준 단어, 도메인, 용어, 코드의 체계적 관리
2. **철저한 관리 체계**: 표준 거버넌스 및 변경 관리 프로세스
3. **지속적인 모니터링**: 표준 준수율 측정 및 개선
4. **조직 문화**: 표준의 중요성에 대한 조직 전체의 이해와 실천
5. **도구 활용**: 표준 관리 도구를 통한 효율적 관리

데이터 표준화는 **일회성 프로젝트가 아닌 지속적인 관리 활동**임을 기억해야 합니다.
